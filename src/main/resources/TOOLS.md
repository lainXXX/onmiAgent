# Agent 工具使用指南

---

## 一、核心原则

1. **技能优先**：处理用户输入时，首先考虑是否有相关的 Skill，即使只有 1% 的可能也要检查
2. **探索先行**：不熟悉代码时，先定位再操作
3. **小步快跑**：复杂任务拆成多个 Task 逐一完成
4. **验证闭环**：每次修改后测试，确保没有破坏现有功能
5. **多次调用**：遇到多任务时，为每个任务单独调用一次工具

---

## 二、工具配合模式

### 2.1 多任务模式

**场景**：用户提出非简单且复杂的任务或目标

```
1. TaskCreate × N  → 每个任务单独调用一次（不要尝试在单次调用中创建多个）
2. TaskUpdate      → 逐个标记为 in_progress
3. 逐个完成子任务
4. TaskUpdate      → 逐个标记为 completed
```

**判断标准**：
- 用户列出多项："帮我完成 A、B、C"
- 用户说"需要做这几件事"
- 任务可拆分为多个独立子目标

**注意：** 如果只有一个琐碎的任务，请不要使用此工具。在这种情况下，直接完成任务效果更好。

---

### 2.2 并行执行模式

**场景**：多个独立任务可以同时执行

```
1. TaskCreate      → 创建主任务
2. TaskCreate × N  → 创建子任务
3. TaskUpdate      → 标记主任务 in_progress
4. Agent (runInBackground: true) × N  → 并行启动多个子 Agent
5. TaskOutput × N  → 获取子任务结果
6. 汇总结果
7. TaskUpdate      → 完成主任务
```

---

### 2.3 耗时操作模式

**场景**：执行耗时的编译、测试、构建操作

```
1. bash (runInBackground: true)  → 后台启动任务
2. TaskOutput (block: true)     → 阻塞等待结果
```

**替代方案**：用 `Agent` 替代 `bash` 处理复杂的多步骤任务

---

### 2.4 探索闭环模式

**场景**：修改代码前先理解代码结构

```
1. glob / grep   → 定位文件
2. read          → 查看内容
3. edit          → 修改
4. bash          → 编译/测试验证
```

**重要**：修改前必须先 read，edit 必须精确替换，不要整体重写

---

### 2.5 知识检索模式

**场景**：回答基于知识库的问题

```
1. semantic_search  → 检索相关文档
2. read_chunks      → 读取具体内容
3. 基于内容回答用户
```

**强制规则**：涉及知识库内容的问题，必须先 semantic_search，不能直接编造

---

### 2.6 用户提问模式

**场景**：需要用户澄清需求或确认方案

```
1. AskUserQuestion  → 发起提问
2. 等待用户响应
3. 根据 answers 继续执行
```

---

## 三、问题处理流程

### 3.1 Bug 修复流程

```
1. 复现问题
   └── bash → 运行触发 Bug 的操作

2. 定位问题
   └── grep → 搜索错误关键词
   └── read → 查看相关代码

3. 分析根因
   └── 结合错误信息和代码逻辑推理

4. 修复问题
   └── edit → 精确修改，不要整体重写

5. 验证修复
   └── bash → 运行测试
   └── 如果失败 → 返回步骤 2
```

**注意**：不要删除或跳过失败的测试，测试失败意味着逻辑错误

---

### 3.2 新功能开发流程

```
1. 规划任务
   └── TaskCreate → 创建任务，明确目标和验收标准
   └── 如果任务复杂 → 拆分为多个子任务

2. 探索现有代码
   └── glob / grep → 了解代码结构
   └── read → 查看相关实现

3. 实现功能
   └── write / edit → 编写代码
   └── 遵循现有代码风格

4. 测试验证
   └── bash → 编译构建
   └── bash → 运行测试

5. 完成收尾
   └── TaskUpdate → 标记任务完成
```

---

### 3.3 代码重构流程

```
1. 理解旧代码
   └── read → 仔细阅读旧代码逻辑
   └── 确认改动范围

2. 创建任务
   └── TaskCreate → 创建重构任务

3. 实施重构
   └── write → 用新实现整体覆盖
   └── 不要逐行修改，容易引入不一致

4. 验证功能
   └── bash → 运行测试确保行为一致
   └── 对比新旧实现的输出
```

---

### 3.4 遇到未知问题时的排查顺序

```
1. 收集信息
   └── 查看错误日志 / 错误信息
   └── bash → 查看当前环境状态

2. 探索代码
   └── glob / grep → 定位相关代码
   └── read → 理解代码逻辑

3. 假设验证
   └── 基于错误信息提出假设
   └── bash → 验证假设

4. 迭代修复
   └── 如果假设正确 → 修复问题
   └── 如果假设错误 → 返回步骤 2，继续探索
```
